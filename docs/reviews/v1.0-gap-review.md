# v1.0 Final Review — Product Consistency & Completeness

> Date: 2026-07-08 · Scope: full implementation (Phases 0–3, U1–U3) audited
> against `SPEC.md` (Draft v0.6) and `PLAN.md`. Method: five parallel audits
> (API/CRD, reconcile/rollout/wiring, git/render/apply, secrets, ship
> readiness), each checking requirement IDs against code with file:line
> evidence, plus a manual pass over release engineering.

## Verdict

The engineering substance is in place: every SPEC §11 field, all six condition
types, both secret backends, both RBAC scopes, the full docs suite, and an
automated A1–A12 acceptance gate on two Kubernetes minors × two install
methods. **No spec feature is missing outright.** The gaps cluster into five
tiers below — one real behavior bug, a handful of spec-vs-code semantic
mismatches to resolve deliberately, observability shortfalls, an unexercised
release process (no v1.0 actually exists as an artifact), and documentation/
spec drift.

## Proposed workstreams (brainstorm)

1. **WS-A — Correctness patch (small, do first).** Fix 1.1 (retract the
   `kohen-stamp` annotation in `finalize`), 1.2 (reset `rolloutInProgress` on
   the `none`/failure paths), 1.3 (skip the workload apply when the built
   apply-object is unchanged — or amend T3's wording if we accept the no-op
   apply). One PR, envtest-covered.
2. **WS-B — Observability round-out.** Emit events for resolution transitions
   and apply/prune; add `kohen_secret_resolve_total{outcome}`; add per-reason
   counters for the WorkloadWired family + `ProgressDeadlineExceeded`; count
   credential-load failures as fetch errors; decide gauge-vs-counter for
   MaxDegradedExceeded. Mostly mechanical.
3. **WS-C — Watch-driven rotation (decide).** Options: (a) metadata-only
   informer (`metadata.k8s.io` partial-object watches) on Secrets +
   ExternalSecrets — keeps memory bounded (the reason the cache was
   restricted) while restoring §6.1/T10 semantics; (b) amend the spec to
   "poll-bounded rotation detection (≤ sync.interval)". (a) is the honest fix
   and not large; (b) is acceptable for 1.0 with (a) as 1.1.
4. **WS-D — Real GitOps e2e.** Add a U3 job installing real Argo CD (SSA mode)
   or Flux and running the A10 scenario; keep the simulated-manager test as
   the fast PR-tier check. Could be nightly-only to protect CI latency.
5. **WS-E — Spec truth-up to v1.0.** Promote SPEC from "Draft v0.6" to a 1.0
   status; close §18 Q1 (`__`); add `ManifestsApplied` + extra reasons to
   §11.4; document `established`/`observedGeneration`/limits/shortname; reword
   R-AUTH.6 to reconcile-time; document 2.5/2.7/2.8 behaviors; state R-AUTH.2
   is post-1.0 and fix the ADR pointer.
6. **WS-F — Release rehearsal & cut.** Make chart-push/signing hard-fail (or
   explicitly gate on configuration); add the namespaced bundle to artifacts;
   cut `v1.0.0-rc.1` to exercise the pipeline end-to-end (multi-arch image,
   chart, SBOM, signature); then make the A12 upgrade test upgrade from that
   rc; align Chart.yaml/SECURITY.md/README/K8s floor; replace LICENSE with the
   full Apache-2.0 text; then tag `v1.0.0`.
7. **WS-G — API version decision (blocking question).** Ship 1.0 serving
   `v1alpha1` with a documented compatibility promise, or promote to
   `v1beta1`/`v1` first? With zero installed users, promotion is the cheapest
   it will ever be (no conversion webhook needed) — recommended before the
   first public tag.

Suggested order: WS-A → (WS-B ∥ WS-C ∥ WS-D) → WS-E → WS-G → WS-F.
